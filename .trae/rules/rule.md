# Handsome Agent 编码规范

> Hermes-Brain + OpenClaw-Body 双核驱动的模块化 AI 智能助手

---

## 核心原则

### 1. 优先使用成熟开源组件

**必须遵循**：优先选择成熟、热门、维护活跃的开源库，避免重复造轮子。

| 场景 | 推荐库 | 原因 |
|------|--------|------|
| 终端菜单 | `inquirer` | 跨平台，成熟稳定 |
| 配置管理 | `pydantic-settings` | 类型安全，验证完善 |
| HTTP 客户端 | `httpx` | 异步支持，现代化 |
| Web 框架 | `fastapi` | 高性能，自动文档 |
| 日志 | `structlog` | 结构化，可扩展 |

**判断标准**：
- GitHub Stars > 1000
- 最近 6 个月有更新
- 有完善的文档和社区支持
- 跨平台支持（如适用）

### 2. 模块化架构

```
adapter/   → 接入层（消息协议转换、鉴权）
brain/     → 决策层（意图识别、任务路由、记忆管理）
core/      → 核心模块（会话、路由、缓存）
shared/    → 共享模块（配置、异常、日志）
executor/  → 执行层（Shell/Docker 执行）
```

**依赖规则**：`shared` 禁止依赖其他模块，`core` 依赖 `shared`，其他层依赖 `core`。

### 3. 安全零信任

- **输入验证**：白名单机制，禁止信任用户输入
- **敏感信息**：从环境变量获取，日志中脱敏
- **路径安全**：防止路径遍历攻击
- **命令执行**：白名单 + 危险模式检测

### 4. 意图理解必须使用 LLM

**必须遵循**：所有涉及用户意图理解、自然语言解析、语义识别的操作，必须优先使用 LLM。

| 禁止做法 | 必须做法 | 说明 |
|---------|---------|------|
| 硬编码关键词字典 | LLM 理解意图 | 用户表达是自然语言，不是固定关键词 |
| 正则表达式匹配意图 | LLM 语义识别 | 意图需要上下文理解 |
| 预定义意图分类 | LLM 动态决策 | 预定义分类无法覆盖所有场景 |

**唯一例外**：当 LLM 不可用时，才可以使用关键词作为降级方案。

### 5. 容错降级

降级链：LLM → 关键词 → 规则 → 默认值

## 代码风格

### 必须遵守

- **PEP 8** 标准
- **缩进**：4 空格
- **行长**：100 字符（软限制 120）
- **编码**：UTF-8
- **导入顺序**：标准库 → 第三方库 → 本地导入

### 命名约定

| 类型 | 规则 | 示例 |
|------|------|------|
| 模块 | `snake_case` | `session_manager.py` |
| 类 | `CamelCase` | `SessionManager` |
| 函数/变量 | `snake_case` | `get_session()` |
| 常量 | `UPPER_SNAKE` | `MAX_RETRY_COUNT` |
| 私有成员 | `_prefix` | `_private_method()` |

## 异常处理

### 异常层次

```
HandsomeAgentError (基类)
  ├── BrainServiceError (2000)
  ├── ExecutorError (3000)
  ├── ToolError (4000)
  └── SecurityError (6000)
```

### 规则

- **禁止裸 except**
- **捕获具体异常**
- **使用异常链**：`raise X from e`
- **日志记录**：ERROR 级别记录异常详情

## 日志规范

### 日志级别

| 级别 | 使用场景 |
|------|----------|
| DEBUG | 开发调试 |
| INFO | 正常流程 |
| WARNING | 异常但可处理 |
| ERROR | 异常且不可处理 |
| CRITICAL | 系统级错误 |

### 规则

- 使用 `logging.getLogger(__name__)`
- 敏感信息脱敏后记录
- 长日志分多行

## 测试规范

### 覆盖率要求

| 模块 | 最低覆盖率 |
|------|------------|
| `core/` | 90% |
| `brain/` | 85% |
| `tools/` | 80% |
| `adapter/` | 75% |
| **整体** | **80%** |

### 测试命名

- 文件：`test_<module>.py`
- 类：`Test<ClassName>`
- 方法：`test_<scenario>`

### 功能必须有测试

**必须遵循**：每个功能点都必须有对应的测试用例。

| 功能场景 | 测试要求 |
|----------|----------|
| 新增函数 | 必须有单元测试 |
| 新增类 | 必须有类级别的测试 |
| 新增 API | 必须有 API 测试 |
| Bug 修复 | 必须有回归测试 |
| 边缘情况 | 必须有边界测试 |

**禁止**：
- 未经测试的功能合并到主分支
- 测试覆盖率低于模块要求

### 模块必须有文档

**必须遵循**：每个模块目录下必须包含 `README.md`，并及时更新。

| 模块 | 必须包含内容 |
|------|--------------|
| **模块介绍** | 功能概述、架构说明 |
| **快速开始** | 安装、配置、基本使用 |
| **API 参考** | 公开接口说明 |
| **示例代码** | 常用场景示例 |
| **变更记录** | 版本更新说明 |

**更新时机**：
- 新增功能 → 必须同步更新 README
- API 变更 → 必须同步更新文档
- Bug 修复 → 可选（重要 Bug 必须记录）
 
### 持续集成（CI）规范

**必须遵循**：每次代码修改完成后，必须确保 CI 检查全部通过。

| 检查项 | 说明 |
|--------|------|
| **CI 必须通过** | 所有 CI 检查项必须通过，包括：单元测试、类型检查、代码格式检查、安全扫描等 |
| **禁止绕过** | 禁止通过修改 CI 配置、跳过测试、禁用检查等方式绕过 CI |
| **失败即失败** | CI 失败意味着代码修改未完成，必须修复问题直至 CI 通过 |
| **本地验证** | 提交前应在本地运行相关检查，确保 CI 能够通过 |

**工作流程**：
1. 本地完成代码修改
2. 运行本地检查（测试、类型检查、格式检查等）
3. 确保所有检查通过
4. 提交代码
5. CI 流水线执行
6. **等待 CI 全部通过** ← 代码修改才算正式完成
7. 如 CI 失败，立即修复问题并重新提交

**禁止**：
- 代码修改完成后不等待 CI 通过就离开
- 提交明知会失败的 CI 配置
- 修复 CI 失败的问题时引入新的问题

## 跨平台兼容

### 必须遵守

- **路径**：使用 `pathlib.Path`，禁止硬编码分隔符
- **系统检测**：`platform.system()` 或 `os.name`
- **命令执行**：使用 `subprocess.run()`，注意平台差异
- **终端交互**：优先使用跨平台库（如 `inquirer`）

```python
from pathlib import Path
config_dir = Path.home() / ".handsome_agent"

IS_WINDOWS = platform.system() == "Windows"
```
